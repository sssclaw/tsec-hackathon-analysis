Long‑Range Agent Design for Complex Tasks

An in‑depth review of the Tencent Cloud Intelligent Penetration Hackathon, winning agent architectures, and best practices for building long‑running AI agents.

1. Introduction

Autonomous agents are software entities powered by large language models (LLMs) that can reason, decide and act. When combined with a collection of tools and a harness that manages state, context and workflows, these agents can tackle tasks far beyond a single query—writing code, planning research, or in this case performing penetration testing. Designing agents for complex, long‑range tasks is challenging: the model’s context window is finite, state must be persisted across multiple operations, and hallucinations or early termination must be detected and corrected. The Tencent Cloud Intelligent Penetration Hackathon was a unique competition that required participants to build fully autonomous agents to hack into vulnerable web applications. Analyzing the top entries provides valuable lessons for constructing industry‑leading agents and harnesses.

2. Event Overview

The second edition of the Tencent Cloud Smart Penetration Hackathon asked teams to create AI agents that autonomously perform penetration tests on a series of web challenges. Human assistance was forbidden after the initial submission. Teams had to manage long‑running sessions, coordinate tool usage, and avoid getting stuck. Evaluation was based on the number of flags submitted, efficiency, stability, and generalisation across unseen challenges. The top teams open‑sourced their solutions, providing a rare window into advanced agent architectures. 3. Agent Basics and Harness Concepts

An agent is an LLM guided by a system prompt, allowed to iteratively observe the environment and act using tools. An agent harness provides the surrounding infrastructure: tool definitions, persistent memory, scheduling logic and safety guards. Effective harnesses must address three core challenges:

  • Finite context. LLMs can read only a limited amount of text; as tasks grow, context windows fill up and earlier information is lost. Solutions include summarisation, hierarchical memory and rewriting strategies, such as the RTK rewrite used by the winning team to compress context into progressively smaller summaries【738928078656857†L28-L99】.
  • Persistent state. Long‑running agents must remember progress across steps or sessions. External storage, configuration files or databases can hold memory (detailed logs and tool outputs) and ideas (high‑level plans) separately, enabling retrieval and reflection without cluttering the working context【738928078656857†L28-L99】.
  • Self‑verification and termination. Without guidance, agents may loop indefinitely or stop prematurely. Harnesses incorporate evaluators or observers that monitor progress and decide when to continue or halt. Addy Osmani describes the need for self‑verification loops and context resets to manage long‑running agents【929632428241154†L0-L33】【929632428241154†L40-L66】, while Anthropic proposes separate planner, generator and evaluator agents to maintain quality over long runs【706319823407743†L41-L47】【706319823407743†L60-L80】.

These concepts underpin most modern multi‑agent frameworks, such as LangGraph, CrewAI, AutoGen and the pi‑coding‑agent SDK, each of which offers distinct memory and orchestration strategies【647580678873974†L40-L96】.

4. Analysis of the Champion Team: BreachWeave

4.1 Architecture

The champion team (ai 小分队) open‑sourced their solution as BreachWeave. The system employs a three‑layer architecture consisting of a Manager, multiple Solver agents, and an Observer【738928078656857†L28-L99】:

  • Manager. Acts as a project manager and global scheduler. It creates challenges, assigns solvers to tasks, monitors progress, summarises state and coordinates the termination of tasks. The manager orchestrates interactions through a set of planning tools that the underlying LLM calls (e.g., planner_start_challenge, planner_launch_solver) to issue structured instructions to the harness.
  • Solver agents. Execute the actual penetration testing tasks: information gathering, vulnerability verification, exploitation and flag submission. Multiple solvers can run concurrently on different attack vectors, maximising coverage. Each solver maintains its own memory and ideas and utilises RTK rewriting to compress long context into succinct summaries for the LLM.
  • Observer. Monitors solver behaviour without interfering. It checks for context drift, early stopping or hallucination and logs issues. The manager consults the observer’s reports to decide whether to continue, restart or stop solvers.

This decoupled design allows specialized agents to operate autonomously while a central orchestrator ensures coherence. The system externalises termination conditions through a Ralph loop that repeatedly builds a state snapshot, runs the planner to decide next actions and executes until a challenge is completed or time limits are reached【738928078656857†L28-L99】.

4.2 Code‑Level and Harness Details

The manager implementation (packages/core/src/challenge/manager.ts) defines constants for timeouts, solver concurrency and memory limits. It exposes methods to start and stop challenges, launch solvers, compress tasks, summarise memory/ideas, and interface with the official API. A notable feature is the Planner Tools registry: a set of declarative functions (e.g., planner_get_state, planner_launch_solver) that the underlying LLM uses to manipulate the challenge state. The manager builds a snapshot containing the current memory, ideas, hint summaries and solver statuses and feeds it into a planner agent session. The planner then decides whether to start a new solver, stop an existing one or submit a flag. By exposing only high‑level operations, the harness constrains the LLM’s actions, ensuring safety and consistency. Helper functions compute digests and format tables to display solver states, while asynchronous loops poll the API for new chay and asynchronous tasks, includes various decoders (base64, JSFuck, PHP filters) and provides a modular toolchain for building payloads【265727408226434†L90-L164】. Compared with BreachWeave, ctfSolver emphasizes specialized modules for each stage of exploitation and heavy use of custom scanners.

5.2 Bytex – Cairn

Bytex’s team openly challenges multi‑agent orthodoxy. Their blog advocates for a blackboard architecture and a single problem‑solving engine instead of separate gatherer/exploiter agents. Cairn uses a directed acyclic graph (DAG) to represent tasks and emphasises minimalism—just one necessary skill for flag submission and no separate knowledge bases or RAG. They argue that general skills or pre‑built modules add overhead and limit adaptability; instead, the agent should learn to use general tools flexibly. This philosophy starkly contrasts with multi‑agent, tool‑heavy designs【106825557809923†L127-L155】.

5.3 Design Patterns from Top Teams

A comprehensive review of the competition highlighted two overarching philosophies: project manager + expert group (manager coordinates multiple specialised agents) and intention‑driven super agent (a single powerful LLM given high‑level intent and allowed to call tools directly). The review distilled six architecture patterns: hierarchical multi‑agent, collaborative multi‑agent, intention‑driven super agent, monitored black‑box agent, client‑server (MCP), and human‑in‑loop supervisor. Successful teams often used sandboxed environments, configuration‑driven tool definitions, long‑context management strategies and frameworks like AutoGen, CrewAI or LangGraph【109715614997539†L220-L256】.

6. Harness Design for Long‑Range Agents

Beyond the hackathon, research from industry sheds light on long‑range agent design. Addy Osmani describes long‑running agents as those that maintain progress across hours or days, requiring context resets, persistent storage and self‑verification loops【929632428241154†L0-L33】. Anthropic’s guide proposes splitting the harness into a planner, generator and evaluator, allowing tasks to be decomposed into chunks with structured hand‑offs and external evaluation to maintain quality【706319823407743†L41-L47】【706319823407743†L60-L80】. These insights align with the hackathon’s top designs: separation of concerns, externalised evaluation and iterative loops. A generic harness for complex tasks should:

  • Isolate memory, ideas and context into separate stores, enabling summaries and rewrites.
  • Expose a minimal set of high‑level planning tools rather than free‑form text instructions to the LLM.
  • Employ observers or evaluators that monitor progress and provide objective feedback.
  • Reset context or compress history periodically to avoid overflow and preserve important information.
  • Plan in loops (e.g., Ralph loop) where the agent iteratively examines state, decides next actions and repeats until termination criteria are met.

Modern agent frameworks offer varying degrees of flexibility and convenience. Key characteristics include memory management, orchestration flexibility and integration ease. For a penetration‑testing or programming agent, the choice depends on trade‑offs:

  • pi‑coding‑agent is ideal when you need maximum control and are willing to build the harness yourself. It offers a minimal core with persistent sessions and encourages the agent to extend itself. BreachWeave demonstrates how to layer a planner and solver on top of Pi, achieving robust performance.
  • LangGraph excels in scenarios requiring complex workflows with many branches and custom memory types, though its complexity may be overkill for simple tasks.
  • CrewAI provides a balanced starting point with built‑in memory stores and role‑based agents; it suits tasks where quick prototyping is more important than fine‑grained control.
  • AutoGen is useful for interactive agents with simpler memory needs; additional evaluation loops must be implemented externally for long‑running tasks.

In summary, for long‑range penetration or programming agents, adopting a customised harness inspired by BreachWeave—built atop pi‑coding‑agent or LangGraph—offers the greatest flexibility to implement context compression, multi‑agent orchestration and external evaluation. However, if rapid development and built‑in features are desired, CrewAI is an excellent alternative.

8. Unknown Unknowns and Additional Insights

When designing long‑range agents, it is easy to underestimate certain pitfalls:

  • Tool reliability. Agents rely on external tools; network errors, rate limits or inconsistent outputs can derail progress. The harness should implement retries and graceful fallbacks.
  • Model variance. Running multiple models improves robustness, but differences in behaviour can produce conflicting plans. A ranking or voting mechanism is needed to resolve disagreements.
  • Security and sandboxing. Penetration agents may accidentally attack unintended targets or leak sensitive data. Always run agents in isolated sandboxes and restrict tool capabilities.
  • Ethical considerations. Autonomous hacking agents raise ethical and legal issues. Such systems should be deployed only for sanctioned assessments with strict safeguards.
  • Evaluation metrics. Standard metrics like number of flags may not capture the quality of reasoning, efficiency or resilience. Consider including logs analysis and human evaluation.

9. Conclusion

The Tencent Cloud Intelligent Penetration Hackathon showcased cutting‑edge approaches to building autonomous agents for complex tasks. The champion BreachWeave system exemplifies a sophisticated harness: separating roles into manager, solver and observer; compressing context via multi‑pass rewriting; externalising termination conditions; and leveraging multi‑model ensembles. Other teams explored minimalistic DAG‑based planners, specialized scanners, and intention‑driven super‑agents, highlighting the diverse design space. Research from industry underscores the importance of long‑running agent loops, persistent state, and external evaluation. Choosing the right framework depends on task complexity and engineering resources: pi‑coding‑agent offers unmatched flexibility for custom harnesses; LangGraph and CrewAI provide structured scaffolds; and AutoGen simplifies tool integration. Ultimately, successful long‑range agents require careful harness design that balances autonomy with safety, enabling LLMs to reason, act and persist over extended missions.
